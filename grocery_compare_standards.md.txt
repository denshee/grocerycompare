# GroceryCompare Development Standards & Principles

**These rules are absolute. Read this document before writing or modifying any code.**

## 1. Zero Cowboy Coding (No Testing in Production)
* NEVER run destructive scripts directly on live data (`Listings` tab).
* ALL backend ETL data transformations must be run on a staging environment (e.g., `Listings_Dev`) first.
* ALL database scripts must use a "Dry Run" flag (`--dry-run`) to output proposed changes to the terminal for user approval before writing a single cell.

## 2. Dynamic Schema Mapping Only (No Hardcoding)
* NEVER hardcode column index numbers (e.g., `row[8]`).
* Database scripts MUST dynamically read row 1 of the sheet, map the headers to their exact indices, and target cells by their column name. 
* If a target column is missing or moved, the script must throw a fatal error and halt gracefully.

## 3. Solid Foundations Over Quick Fixes
* No "proof of concepts" or hacky regex string-matching that only targets a fraction of the database.
* Build enterprise-grade, edge-to-edge solutions. If we build a parser, it must logically handle all 10,000 items, Fences, and edge cases before deployment.

## 4. Protect Existing Assets
* Destructive overwrites to critical infrastructure (like `Image_URL` columns) must be mathematically impossible. Define strict "Read-Only" and "Write-Only" boundaries in the code.
